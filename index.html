01 Problem Illustration

Scenario: The 2014-Inspired Kelantan Flood Crisis

Kuala Krai, a district in Kelantan, Malaysia, is one of the most flood-prone regions in Southeast Asia. During the catastrophic 2014 monsoon season, floodwaters reached unprecedented levels, displacing over 200,000 people and devastating villages along the Kelantan River. The disaster exposed critical weaknesses in supply logistics — with limited transport capacity, relief agencies faced impossible choices about which supplies to send first.

The Optimization Challenge

A relief truck with a 1,000 kg capacity is dispatched from the Kuala Krai coordination center. There are 20 different supply packages available, each with a specific weight and relief value score (calculated from population impact, urgency, and medical necessity). The total weight of all packages (1,515 kg) exceeds the truck's capacity — so we cannot take everything.

Goal: Maximize the total relief value delivered to affected villages (Kg. Manek Urai, Kg. Dabong, Kg. Bertam, Kg. Laloh) within the truck's weight constraint of 1,000 kg.

Why Finding the Optimal Solution Matters

Lives at Stake

Suboptimal allocation means critical medical supplies or clean water may be left behind, directly risking lives in isolated villages.

Resource Scarcity

During disasters, transport vehicles and fuel are extremely limited. Each trip must deliver maximum impact.

Time Pressure

Flood victims face dehydration and disease within 48 hours. An optimal algorithm ensures fast, correct decisions under pressure.

02 Algorithm Paradigm

Why Dynamic Programming?

This problem is a classic 0/1 Knapsack Problem: each supply package is either loaded (1) or left behind (0), and we must maximize total relief value without exceeding the weight capacity. Dynamic Programming is the ideal paradigm because it exhibits both optimal substructure and overlapping subproblems.

Optimal Substructure

The optimal solution for n items with capacity W contains the optimal solution for (n-1) items. If item i is included, the remaining items must be optimally selected for capacity (W - weight_i).

Overlapping Subproblems

When deciding for each item whether to include or exclude it, many subproblems dp[i][w] are computed repeatedly in a naive recursive approach. DP stores these in a table, computing each only once.

Recurrence Relation

dp[i][w] = max( dp[i-1][w] , value[i] + dp[i-1][w - weight[i]] )

For each item i and capacity w, we take the maximum of two choices: (1) exclude item i and keep the value from the previous items, or (2) include item i and add its value to the optimal solution for the remaining capacity.

Optimization Function

maximize ∑ (value_i × x_i)    subject to    ∑ (weight_i × x_i) ≤ W,   x_i ∈ {0, 1}

Base Case

dp[0][w] = 0 for all w ≥ 0  —  With zero items, the maximum value is zero regardless of capacity.

03 Pseudocode

// 0/1 Knapsack - Bottom-Up Dynamic Programming
// Input:  packages[] of (weight, value), capacity W
// Output: maximum relief value and selected items

FUNCTION SolveKnapsack(packages[], n, W):
    // Create DP table of size (n+1) x (W+1)
    CREATE dp[0..n][0..W] initialized to 0

    // Fill table bottom-up
    FOR i = 1 TO n:
        FOR w = 0 TO W:
            // Option 1: Don't include item i
            dp[i][w] = dp[i-1][w]

            // Option 2: Include item i (if it fits)
            IF packages[i].weight ≤ w:
                includeVal = packages[i].value + dp[i-1][w - packages[i].weight]
                dp[i][w] = MAX(dp[i][w], includeVal)

    RETURN dp[n][W]  // optimal value


FUNCTION Traceback(dp[][], packages[], n, W):
    selected = empty list
    w = W

    FOR i = n DOWNTO 1:
        IF dp[i][w] ≠ dp[i-1][w]:
            // Item i was included in optimal solution
            ADD i to selected
            w = w - packages[i].weight

    RETURN REVERSE(selected)

04 Algorithm Comparison

We reviewed five major algorithm paradigms for suitability to this problem:

Paradigm Strengths Weaknesses Suitable?
Sorting Simple, fast O(n log n). Good for ordering items by priority. Sorting alone doesn't solve the constraint optimization. Only helps as a preprocessing step. Insufficient
Divide & Conquer Breaks problem into subproblems. Works well for problems like merge sort, binary search. Subproblems overlap heavily in knapsack; D&C recomputes them, leading to O(2^n) without memoization. Inefficient
Dynamic Programming Guarantees optimal solution. Handles overlapping subproblems via tabulation. Pseudo-polynomial O(nW). Space O(nW) can be large for very high capacity values. Not truly polynomial (pseudo-polynomial). SELECTED
Greedy Fast O(n log n). Simple to implement. Optimal for fractional knapsack. NOT optimal for 0/1 knapsack. May miss better combinations. Our test shows it missed 5 value points. Not Optimal
Graph Algorithms Excellent for network/routing problems (shortest path, MST). Problem lacks graph structure (no edges/nodes). Would require artificial graph construction with no benefit. Unsuitable
Conclusion: Dynamic Programming is the most suitable paradigm. It guarantees optimality, handles the constraint naturally via the DP table, and runs in O(nW) time — fast enough for real-time disaster response.

05 Chosen Approach: Dynamic Programming

Final Selected Solution: After evaluating all five algorithm paradigms — Sorting, Divide & Conquer, Dynamic Programming, Greedy, and Graph Algorithms — we selected Dynamic Programming (Bottom-Up Tabulation) as the final solution for the flood relief supply allocation problem.

Why Dynamic Programming is the Final Selected Solution

Dynamic Programming is chosen because it is the only paradigm that satisfies all three critical requirements of our disaster relief scenario: guaranteed optimality (every value point matters when lives are at stake), practical efficiency (O(nW) runs in milliseconds for real-world input sizes), and traceback capability (we can identify exactly which supply packages to load, not just the total value).

How It Solves the Problem

The algorithm constructs a 2D table of size (n+1) × (W+1), where n = 20 items and W = 1,000 kg. Each cell dp[i][w] stores the maximum relief value achievable using the first i items with a capacity limit of w kg. The table is filled bottom-up using the recurrence relation, and the final answer is found at dp[20][1000]. A traceback through the table then recovers the exact set of selected packages.

06 Java Implementation

The complete Java program implements three algorithms for comparison: DP (primary), Greedy, and Brute Force.

Core DP Method

public static int[][] solveKnapsack(SupplyPackage[] packages, int capacity) {
    int n = packages.length;
    int[][] dp = new int[n + 1][capacity + 1];

    // Base case: dp[0][w] = 0 for all w (no items = no value)

    // Fill the DP table bottom-up
    for (int i = 1; i <= n; i++) {
        for (int w = 0; w <= capacity; w++) {
            dp[i][w] = dp[i - 1][w];  // exclude item i

            if (packages[i - 1].weight <= w) {
                int includeValue = packages[i - 1].reliefValue
                    + dp[i - 1][w - packages[i - 1].weight];
                dp[i][w] = Math.max(dp[i][w], includeValue);
            }
        }
    }
    return dp;
}

Traceback Method

public static List<Integer> traceback(int[][] dp, SupplyPackage[] packages, int capacity) {
    List<Integer> selected = new ArrayList<>();
    int n = packages.length;
    int w = capacity;

    for (int i = n; i > 0; i--) {
        if (dp[i][w] != dp[i - 1][w]) {
            selected.add(i - 1);
            w -= packages[i - 1].weight;
        }
    }
    Collections.reverse(selected);
    return selected;
}

08 Algorithm Analysis

Correctness Proof

The algorithm is correct by induction on the number of items considered.

Base case: dp[0][w] = 0 for all w. With zero items, the maximum value is correctly zero.

Inductive step: Assume dp[i-1][w] correctly stores the optimal value for items 1..i-1 with capacity w. For item i, there are exactly two possibilities: (1) item i is not in the optimal solution — then dp[i][w] = dp[i-1][w], which is correct by hypothesis; (2) item i is in the optimal solution — then dp[i][w] = value[i] + dp[i-1][w - weight[i]], which is correct because the remaining items must be optimally chosen for the remaining capacity (optimal substructure). Taking the max of both options guarantees correctness.

Termination: The outer loop runs exactly n iterations, the inner loop runs exactly W+1 iterations. The algorithm always terminates.

Time Complexity

CaseComplexityExplanation
Best Case Θ(nW) The entire table must always be filled regardless of input. No early termination possible.
Average Case Θ(nW) Same as best case — every cell is computed exactly once via the nested loops.
Worst Case Θ(nW) Same as best case. The algorithm is deterministic and input-independent in its loop structure.

For our scenario: Θ(20 × 1000) = Θ(20,000) operations — trivially fast for modern hardware.

Space Complexity

Θ(nW) for the DP table. Can be optimized to Θ(W) using a 1D rolling array, but this sacrifices the ability to traceback which items were selected.

Growth of Function Comparison

DP is the sweet spot: it guarantees optimality like brute force but with polynomial-like growth for practical W values. Greedy is faster but sacrifices correctness. Brute force is infeasible beyond ~25 items.

Recurrence Analysis (Master Theorem Not Applicable)

The recurrence T(n, W) = T(n-1, W) + T(n-1, W-w_i) + O(1) without memoization leads to exponential time. DP eliminates this by storing each subproblem result in the table, reducing the total number of unique subproblems to n × W, each solved in O(1). This is why DP achieves O(nW) — pseudo-polynomial time.

09 Reflection & Learning Outcomes

How Our Understanding Changed

When we first encountered the topic of algorithm paradigms in CSC4202, we viewed them as isolated techniques — sorting is for ordering, greedy is for quick decisions, DP is for table-filling. This project changed that perspective entirely. By modelling a real disaster scenario and testing multiple approaches against each other, we learned that choosing the right paradigm is itself a design decision with real consequences. The greedy algorithm's 5-point shortfall might seem small on paper, but in the context of flood relief it could mean a village missing critical medicine.

"Before this project, I thought greedy algorithms were 'good enough' for most problems. Running it against DP on our flood data and seeing it miss the optimal solution by 5 value points made me realise that 'close to optimal' can still mean life-or-death consequences in real applications." — Fawzia Moradi, Algorithm Design & Analysis

Our Journey: Week by Week

Week 10 — Problem Definition

We brainstormed several disaster scenarios and settled on the Kelantan flood because of its personal relevance — some of us had family affected by the 2014 floods. Translating a human crisis into mathematical variables (weights, values, constraints) felt uncomfortable at first, but we realised that formalising the problem is exactly how algorithms can help save lives.

Week 11 — Algorithm Selection

We debated extensively between greedy and DP. The greedy approach was tempting because of its simplicity and speed. It was only after we built a small test case with 6 items and found that greedy missed the optimal combination that we committed to DP. This hands-on comparison taught us more about the greedy-choice property (and its limitations) than any textbook example had.

Week 12 — Implementation

Translating the recurrence relation into working Java code was more challenging than expected. Our first implementation had an off-by-one error in the traceback function that caused it to skip the first item. Debugging this forced us to truly understand how the DP table indices map to the original item array — a detail that's easy to overlook when reading pseudocode.

Week 13 — Analysis & Testing

Writing the correctness proof by induction was the most intellectually demanding part. We had to reason carefully about why the max of include/exclude truly captures the optimal decision. Running all three algorithms and comparing execution times made the theoretical complexity classes (O(nW) vs O(2n)) feel concrete — watching brute force take 28× longer on just 20 items drove home why exponential growth is impractical.

Week 14 — Portfolio & Presentation

Building the portfolio pushed us to explain our work clearly enough for someone outside the course to understand. This process revealed gaps in our own understanding — if we couldn't explain a concept simply, we hadn't fully grasped it ourselves. The portfolio became a learning tool, not just a submission requirement.

Connecting to Previous Learning

This project drew on concepts from across the entire CSC4202 syllabus. The sorting algorithms we studied in early weeks appeared as a preprocessing step in our greedy approach (sorting items by value-to-weight ratio). The divide-and-conquer paradigm helped us understand the recursive structure of the knapsack problem before we applied memoization to convert it into DP. Even graph algorithms, which we concluded were unsuitable for this problem, informed our analysis — understanding why a paradigm does not fit is as valuable as knowing when it does.

Beyond CSC4202, we applied skills from our programming courses (Java, object-oriented design), discrete mathematics (proof by induction), and even data structures (2D arrays, ArrayLists, sorting with Comparators). The project showed us that algorithm design is not an isolated discipline — it sits at the intersection of mathematics, programming, and real-world problem solving.

Key Takeaways

Optimality Matters

A "nearly correct" answer is not acceptable when lives depend on it. DP's guarantee of optimality is what makes it the right choice for critical resource allocation, even if greedy is faster.

Theory Meets Practice

The gap between understanding a recurrence on a whiteboard and implementing it in bug-free code is significant. This project bridged that gap and made us better programmers and analysts.

Collaboration is Essential

Dividing work across algorithm design, implementation, and documentation forced us to communicate precisely about technical concepts. Explaining DP to a teammate who then had to code it caught errors early.

Future Applications

This project has expanded our view of where algorithm design applies beyond the classroom. The 0/1 Knapsack model could be extended to multi-truck fleet allocation (multiple knapsacks), time-dependent relief where item values change as hours pass (dynamic value functions), or multi-objective optimisation balancing value, coverage area, and fairness across villages. We are now more confident in identifying real-world problems that can be modelled as optimisation problems and selecting the appropriate algorithmic paradigm to solve them.
